1 · Overview & the core idea

What the system does, and the one design decision everything else follows from.

A customer talks to the agent in a chat window. The agent looks up the customer's orders, checks the refund request against a written policy, and resolves it to exactly one outcome: approved denied escalated (or pending if it refuses without a formal record). An admin dashboard shows the agent's internal reasoning, every tool call, token cost, latency, and a human-review queue.

Core idea — defense in depth. The LLM proposes; a deterministic Python policy engine disposes. The submit_refund tool re-validates every request against the policy in code before writing anything — so even a fully jailbroken model that is argued into calling submit_refund on a final-sale item gets a denied back. The prompt is the first line of defense; the policy engine is the non-bypassable second.

2 · System architecture

Four layers with one responsibility each. The agent only touches the world through tools; enforcement lives in code, not in the prompt.

Separation of concerns: UI (frontend/) ↔ API (app/api) ↔ orchestration (app/agent) ↔ domain logic (core/policy.py) + data (app/db). The clearest proof the layers are decoupled: swapping the LLM provider (Anthropic → OpenAI) touched only config, nodes, pricing, and dependencies — the agent, policy, DB, API, and entire frontend were untouched.

4 · Data pipeline (synthetic data → database)

The challenge asks for LLM-generated mock data. Here's how it flows from text files into a queryable database the agent reasons over.

Generate (LLM). Two artifacts, authored by an LLM: app/data/refund_policy.md (the binding rules) and app/data/crm_seed.json (15 customers / 17 orders, deliberately covering every policy branch).
Relative dates. Orders store *_days_ago offsets (e.g. delivered_days_ago: 5) rather than absolute dates, so window-based scenarios (29 vs 31 days) stay valid whenever the DB is rebuilt.
Seed (app/db/seed.py). Converts offsets → timestamps, computes each order total from its items, and inserts customers → orders → items → (optional) pre-existing refunds via the ORM.
Auto-seed on boot. app/main.py lifespan seeds the DB on first start if it's empty — zero-config: docker compose up just works.
Query at runtime. The agent's tools read this data through SQLAlchemy; decisions are written back to refunds and the full run to agent_runs.

The 15 profiles are engineered as a test matrix: happy path, final-sale item, gift card & perishable (non-returnable), expired window (60d) and the 29/31-day boundary, not-yet-delivered, already-refunded, a $500-exactly vs $529 boundary, a mixed final-sale order, and an over-$500 VIP escalation.

6 · The agent — a LangGraph state machine

A small, explicit graph. The model reasons and asks for tools; a tool node runs them and loops back, until the model gives a final answer.

        ┌──────────────────────────── AgentState ────────────────────────────┐
        │  messages: list   (add_messages reducer — keeps tool/AI history)    │
        │  iterations: int  (loop guard, capped at MAX_AGENT_ITERATIONS=8)    │
        └─────────────────────────────────────────────────────────────────────┘

   START ──▶ ( agent node )                       make_agent_node / make_tool_node
                  │  llm_with_tools.invoke(messages)        in app/agent/nodes.py
                  │  → records tokens·cost·latency·reasoning to the Tracer
                  ▼
            should_continue ?
              ├── has tool_calls and iterations < cap ──▶ ( tools node ) ──┐
              │        executes each tool, records I/O + retries           │
              │        appends ToolMessage results                         │
              │                                                            ▼
              │                                                      (back to agent)
              └── final answer (no tool_calls) ─────────────────────────▶  END

Why a graph (not a raw while-loop or CrewAI)?

The graph makes the control flow and state explicit and inspectable, gives first-class hooks for streaming and per-node instrumentation, and keeps the security boundary tiny — the agent's only way to affect anything is by emitting a tool call that the tool node executes. A multi-agent framework would be heavier than a single refund agent needs.

Tools, the model, and the tracer are built per request via a RunContext (DB session, the authenticated customer id, settings, tracer). That's how identity is injected server-side rather than trusted from the model.

7 · Tools — the agent's hands

Built by a factory (app/agent/tools.py) that closes over the authenticated identity. The model never supplies a customer id; the tools act as the signed-in customer.

ToolWhat it does
get_account_summary()the authenticated customer's profile.
get_orders()that customer's orders (no customer arg — identity injected).
get_order_details(order_number)items + final-sale/returnable flags + refund status; returns "not found" for an order the customer doesn't own (no data leak).
get_refund_policy()the written policy text, for grounding.
check_refund_eligibility(order_number, amount?, item_sku?)advisory call into the policy engine — returns action + the rule that applies, writes nothing.
submit_refund(order_number, amount?, item_sku?, reason)the system of record — re-validates against the policy engine, then writes the decision. Refuses forbidden refunds regardless of the conversation.
escalate_to_human(order_number, reason)records an escalation ticket for the human-review queue.

8 · The policy engine — the law (deterministic)

core/policy.py is a pure function over plain facts — no LLM, no network, no ORM. Both check_refund_eligibility and submit_refund route through it.

evaluate_refund(...) returns a PolicyDecision(action, rule, explanation, refundable_amount), evaluating rules in this order (first match wins):

#RuleOutcome
1order not owned by the authenticated customerdeny identity_mismatch
2order already has an approved refunddeny duplicate_refund
3any targeted item is final saledeny final_sale
4any targeted item is non-returnable (gift card, perishable…)deny non_returnable
5not delivereddeny not_delivered
6outside the 30-day return windowdeny window_expired
7requested amount > refundable amountdeny amount_exceeds_refundable
8otherwise eligible but amount > $500escalate over_threshold
9otherwiseapprove eligible

Boundaries are exact and tested: $500.00 approves, $500.01 escalates; delivered 29 days ago approves, 31 days denies. Thresholds come from settings (RETURN_WINDOW_DAYS, ESCALATION_THRESHOLD_USD).

9 · Request lifecycle (end to end)

What happens on POST /api/chat, from HTTP to a recorded decision.

Middleware. A request_id is attached and echoed on the response (X-Request-ID); errors are returned as a structured envelope.
Idempotency. If an Idempotency-Key header was seen before, the cached response is replayed — the agent does not run again.
Rate limit. A per-IP sliding-window limiter (20/min) guards the endpoint; over the limit → 429 with Retry-After.
Identity + guardrail. The customer is resolved from the signed-in email; the message is screened by the heuristic injection detector and the result is flagged on the run.
Compose prompt. A system prompt is built with the authenticated identity and the binding policy; prior history + the new message are appended.
Run the graph. build_agent(ctx).invoke(...) drives agent ⇄ tools until a final answer; the Tracer records every LLM turn and tool call.
Enforcement. Any submit_refund re-checks the policy engine and writes the decision to refunds — the binding step.
Persist + respond. The full trace + totals are written to agent_runs; the API returns the reply, decision, run id, and a summary. If an idempotency key was supplied, the response is cached.

10 · Streaming (Server-Sent Events)

POST /api/chat/stream gives the chat a production-grade, real-time feel.

A streaming variant of the loop (app/agent/streaming.py) reuses the same tools, policy enforcement, and tracer, but is unrolled so it can emit events as they happen. The browser parses the SSE frames (the Next.js client in lib/api.ts) and fills the bubble token-by-token while showing live tool status ("Reviewing the order…").

event: start        → {session_id, request_id, injection_flagged}
event: tool_start   → {name:"get_order_details", input:{order_number:"LP-1001"}}
event: tool_result  → {name:"get_order_details", ok:true, output:{…}}
event: token        → {text:"Your"}  {text:" refund"}  {text:" has"} …
event: done         → {decision:"approved", run_id, summary:{tokens, cost, latency}}

The non-streaming graph remains the path used by the JSON API, the eval harness, and the tests, where precise token accounting is cleanest.

11 · Observability & cost

Two complementary layers, for two audiences.

In-app Tracer (the product surface)

core/observability.py records, per run: each LLM turn (input/output tokens, cache tokens, cost from core/pricing.py, latency, stop reason, the model's reasoning, requested tool calls) and each tool step (input, output, latency, ok/error, retry). Persisted to agent_runs.trace_json and replayed in the /admin trace viewer — failed/retry steps highlighted in red.

LangSmith (the engineer view)

Set LANGSMITH_API_KEY + LANGSMITH_TRACING=true and every LLM call, tool call, and LangGraph node is traced to your LangSmith project — prompts, tokens, latencies, and the full graph, with zero code change (it activates from env vars).

Cost is computed live from a per-model price table (gpt-4o = $2.50 / $10.00 per 1M in/out tokens), so the dashboard's dollar figure is real, not estimated.

12 · Security & reliability (edge layer)

ConcernImplementation
No double refundsIdempotency-Key replay (idempotency_keys table) — a retry returns the cached run, and the policy's duplicate rule blocks a second approval anyway.
AbusePer-IP rate limiting (in-memory sliding window; Redis for multi-instance prod).
TraceabilityRequest IDs on every response + a structured error envelope ({detail, error{type,message}, request_id}).
Prompt injectionHeuristic guardrail (core/security.py) flags "ignore previous instructions", "developer mode", authority/fake-policy claims — observe & log (an ⚠ badge + an "injection attempts" stat), not block, since the policy engine already makes attacks harmless and blocking risks false-positive lockouts.
IdentityInjected server-side via RunContext — impersonation is structurally impossible and other customers' data is never revealed.
Why the guardrail observes instead of blocks: the binding defense is the deterministic engine. The detector turns attacks into a measurable signal (visible on the dashboard and counted in the evals) without degrading legitimate users.

13 · Human-in-the-loop

"Refunds over $500 require human escalation" is only complete if a human can act.

Over-threshold refunds are recorded as escalated and surface in an admin review queue (GET /api/escalations). A reviewer approves or denies (POST /api/escalations/{id}/resolve); the refund is updated with decided_by = human and a resolved_at timestamp. The dashboard shows a live "awaiting review" count.

14 · Evaluation harness — measuring resilience

"Agent resilience" is only credible if it's a number. backend/evals/ runs the real agent against a labeled set and scores it.

23 scenarios across four categories — happy, edge (every denial rule + boundaries), escalation, and adversarial (developer-mode, CEO authority, threats, fake "another agent approved it", fake policy update, cross-customer impersonation, system-prompt exfiltration, multi-turn pleading). The runner reports:

MetricMeaning
Decision accuracyacceptable final decision, overall + per category.
Policy-violation rateunauthorized approvals on adversarial prompts. Target 0 — the headline gate.
Held-the-line rateshare of attacks the agent did not approve.
Escalation accuracyof refunds that should escalate, how many did.
Cost / latencytotal + average per scenario, p95.
Judgean LLM-as-judge grades reply quality (tone, holding the line, no prompt leak).

The runner exits non-zero on any violation, so a CI job can gate on resilience. Validated live: 0 violations · 100% held-line · judge 5/5 on the adversarial subset.

16 · Deployment

One command locally; a free path to the cloud.

Local: docker compose up --build brings up backend (8000) + frontend (3000); the backend auto-seeds. Cloud (free): backend as a Render Docker web service (render.yaml blueprint; binds the host-injected $PORT), frontend on Vercel (set BACKEND_URL). The Next.js /api proxy means the browser only talks to Vercel — no CORS. Full guide in DEPLOY.md.
